LDAP Server Deployment & Splunk Integration Guide

This guide covers how to deploy a lightweight LDAP server on AWS EC2, understand the authentication flow, and configure Splunk integration.

Theory: How LDAP Authentication Works in Splunk

Before deploying, it's important to understand what happens when a user logs into Splunk using LDAP:

  1. Bind (Connection): Splunk connects to the LDAP server using a service account (Bind DN/Password) to verify it can read the directory.
  2. Search: When a user like salesuser1 types their credentials, Splunk searches the directory (Base DN) to find that user entry.
  3. Verify Password: Splunk attempts to "bind" again, this time as the salesuser1 using the password they entered. If successful, the user is authenticated.
  4. Retrieve Groups: Splunk finds all LDAP groups the user belongs to (e.g., SalesUserGroup).
  5. Map Roles: Splunk looks at its configuration to see if SalesUserGroup maps to a Splunk Role (e.g., user role).
  6. Grant Access: If a match is found, the user is logged in with that specific role.

Step 1: Deploy LDAP Server on EC2

We have automated the complete installation process. You only need to run one command on your Amazon Linux EC2 instance.

1. Connect to your EC2 Instance

ssh -i "your-key.pem" ec2-user@<your-ec2-ip-address>

2. Run the Auto-Install Command

Copy and paste this entire line into your terminal:

sudo bash -c 'yum update -y; yum install git -y; git clone https://github.com/splunklearners1/adminldap.git; cd adminldap; chmod +x autoldap.sh; ./autoldap.sh'

This script automatically:

  • Installs Git, Docker, and Docker Compose
  • Clones the LDAP project repository
  • Builds the Docker container
  • Starts the LDAP service on port 1389
  • Starts the Web UI on port 3000
Verify Installation:
Open your browser to: http://<your-ec2-ip>:3000
If you see the LDAP User Manager dashboard showing active users and groups, the server is running!

Step 2: Configure Splunk Integration

You can configure Splunk using either the configuration file (Recommended) or the Web UI.

Method A: The "Config File" Way (Recommended)

This method prevents human error. Copy the content below and save it as authentication.conf. 

[authentication]
authType = LDAP
authSettings = LocalLDAP

[roleMap_LocalLDAP]
admin = SplunkAdmins
power = SplunkPowerUsers
user = SalesUserGroup;SplunkUsers

[LocalLDAP]
SSLEnabled = 0
anonymous_referrals = 0
bindDN = cn=ldapadmin,ou=users,o=splunk
bindDNpassword = password
charset = utf8
groupBaseDN = o=splunk
groupBaseFilter = (objectClass=groupOfNames)
groupMappingAttribute = dn
groupMemberAttribute = member
groupNameAttribute = cn
host = YOUR_EC2_IP
nestedGroups = 0
network_timeout = 20
port = 1389
realNameAttribute = cn
userBaseDN = o=splunk
userNameAttribute = uid
emailAttribute = mail

Next Steps:

  1. Paste the copied content into a new file named authentication.conf.
  2. IMPORTANT: Change host = YOUR_EC2_IP to your actual EC2 Private IP.
  3. Move the file to: $SPLUNK_HOME/etc/system/local/
  4. Restart Splunk: $SPLUNK_HOME/bin/splunk restart

Method B: The "Splunk Web UI" Way (Manual)

If you prefer using the Splunk Interface:

  1. Go to Settings > Authentication Methods.
  2. Select LDAP and click Configure Splunk to use LDAP.
  3. Add a new strategy with these exact settings:
    Setting Value
    Host YOUR_EC2_IP
    Port 1389
    Bind DN cn=ldapadmin,ou=users,o=splunk
    Password password
    User Base DN o=splunk
    User Name uid
    Real Name cn
    Email mail
    Group Base DN o=splunk
    Static Member Attr member
    Group Name Attr cn
  4. Map Groups: Click "Map groups" and link them:

Step 3: Test User Authentication

Now, verify that the integration is working by logging in as an LDAP user.

1. Test Credentials

Use these pre-configured users to test different access levels:

Username Password LDAP Group Splunk Role
salesuser1 password123 SalesUserGroup user (Basic Access)
ldapadmin password SplunkAdmins admin (Full Access)

2. Verification Steps

  1. Open Splunk in an Incognito/Private window (to avoid cached sessions).
  2. Enter Username salesuser1 and Password password123.
  3. Click Login.
  4. Success Check: If logged in successfully, click your name in the top bar. You should see "user" role listed.

Troubleshooting

  • Connection Refused? ensure your EC2 Security Group allows Inbound Traffic on port 1389 (LDAP) and 3000 (Web UI).
  • Invalid Credentials? Check that you are typing the password correctly. Passwords are case-sensitive.
  • User logged in but no access? This means the "Role Mapping" step was missed. Go back to Method B, Step 4.
